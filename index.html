Brand Copy-Paste error ('Alpha Trophies') and Trailing Hyphen bug in titles

The homepage and product meta descriptions reference competitor/sister brand 'Alpha Trophies' instead of 'Olympia Trophies'. All title tags end with a trailing hyphen due to layout interpolation issues.

Page: https://beta.olympia.com.au/

SEO Audit Finding

Severity: High Channels: SEOAEO

Confidence 99/100 evidence: Homepage and Product page DOM meta tags

The homepage and product page meta descriptions and Twitter descriptions contain a critical brand copy-paste error: 'Discover excellence at Alpha Trophies – Australia's premier online destination...'. This associates Olympia's web search snippets with a completely different brand name, harming CTR and brand entity association.

Additionally, the title tags on the Homepage and Product pages render a trailing hyphen: Shop Online at Australia's Largest Trophies, Medals, Plaques & Custom Corporate Awards Shop - . This occurs because the Laravel template tries to append a site name variable (like $siteName) which is empty or unpassed.

class="cm"><!-- Meta description (Homepage + Product layout) -->
class="kw"><meta name=class="str">"description" content="Discover excellence at Alpha Trophies – Australia's premier...">

class="cm"><!-- Title tag interpolation in layout -->
class="kw"><title>Shop Online at Australia's Largest ... Awards Shop - {{ $siteName }}class="kw"></title>
Revamp Implementation

Update the Laravel layout metadata config. Replace 'Alpha Trophies' with 'Olympia Trophies'. Fix the title tag generator in the Blade layout to ensure the brand name 'Olympia Trophies' is correctly appended without trailing symbols.

class="cm"><!-- Meta description -->
class="kw"><meta name=class="str">"description" content="Discover excellence at Olympia Trophies – Australia's premier online destination for Trophies, Medals, Plaques, and Custom Corporate Awards.">

class="cm"><!-- Title tag -->
class="kw"><title>Buy Trophies &amp; Medals Online | Olympia Trophiesclass="kw"></title>

3
On-Page

Product schema warnings — shippingDetails and returnPolicy at root instead of offers

The newly added Product schema includes return policy and shipping details. However, they are defined at the root level of the Product object rather than inside the Offer block where search engines require them.

Page: https://beta.olympia.com.au/product/hero-shield-basketball-silver-gold-trophy

SEO Audit Finding

Severity: Medium Channels: SEO

Confidence 95/100 evidence: Product page JSON-LD block

The product page now has a valid Product JSON-LD block. However, shippingDetails and hasMerchantReturnPolicy are placed at the root level of the Product schema. Google's Merchant Listings structured data validator requires these properties inside the Offer or AggregateOffer object.

Because they are not correctly nested inside the offers block, Google Shopping crawler fails to extract them, generating structured data warnings in Search Console and blocking merchant rich snippets (price, stock, delivery badges) in search results.

{
  "@context": "https://schema.org",
  "@type": "Product",
  "name": "Hero Shield-Basketball (Silver-Gold) Trophy",
  "offers": {
    "@type": "AggregateOffer",
    "lowPrice": "12.90",
    "highPrice": "17.65"
    // offers block is closed here
  },
  "hasMerchantReturnPolicy": { ... }, // ❌ Root property
  "shippingDetails": { ... }           // ❌ Root property
}
Revamp Implementation

Move the hasMerchantReturnPolicy and shippingDetails blocks inside the offers (Offer or AggregateOffer) block in your product schema JSON generator template.

{
  "@context": "https://schema.org",
  "@type": "Product",
  "name": "Hero Shield-Basketball (Silver-Gold) Trophy",
  "offers": {
    "@type": "AggregateOffer",
    "priceCurrency": "AUD",
    "lowPrice": "12.90",
    "highPrice": "17.65",
    "offerCount": "7",
    "shippingDetails": {
      "@type": "OfferShippingDetails",
      "shippingRate": { "@type": "MonetaryAmount", "value": "0", "currency": "AUD" },
      "shippingDestination": { "@type": "DefinedRegion", "addressCountry": "AU" }
    },
    "hasMerchantReturnPolicy": {
      "@type": "MerchantReturnPolicy",
      "applicableCountry": "AU",
      "returnPolicyCategory": "https://schema.org/MerchantReturnFiniteReturnWindow",
      "merchantReturnDays": "14",
      "returnMethod": "https://schema.org/ReturnByMail",
      "returnFees": "https://schema.org/FreeReturn"
    }
  }
}

How to verify: Paste a product page URL into https://search.google.com/test/rich-results and verify Merchant Listings has 0 warnings for shipping or returns.

Conflicting robots.txt directives — Cloudflare managed block vs custom allow rules

The custom rules correctly allow AI retrieval crawlers (ChatGPT-User, Claude-SearchBot, PerplexityBot), but the Cloudflare Managed Content directives disallowing them remain at the top. This can cause conflicting bot parsing.

Page: https://beta.olympia.com.au/robots.txt

SEO Audit Finding

Severity: Medium Channels: GEOSEO

Confidence 95/100 evidence: robots.txt at https://beta.olympia.com.au/robots.txt

The live robots.txt contains the recommended custom rules that explicitly permit AI search crawlers (ChatGPT-User, Claude-SearchBot, and PerplexityBot) with Allow: /. This resolves the AI citation block for RAG grounding.

However, the Cloudflare Managed Content block at the top of the file remains active, issuing Disallow: / for GPTBot, ClaudeBot, Google-Extended, and Applebot-Extended. Because crawlers parse directives differently (some read first match, some read most specific), having duplicate, conflicting rules for the same user-agents can lead to indexing exclusion.

Specifically, Google-Extended and Applebot-Extended are disallowed in the Cloudflare block but allowed in the custom block below. This needs to be resolved by disabling the Managed Content block in Cloudflare dashboard to ensure single, clear directives.

# BEGIN Cloudflare Managed content
User-agent: Google-Extended
Disallow: /
# ...
# END Cloudflare Managed Content

# Custom Block:
User-agent: Google-Extended
Allow: /
Revamp Implementation

Disable Crawler Hints -> Managed Robots.txt in the Cloudflare dashboard. Maintain only the custom, verified robots.txt directives that explicitly block training scrapers (CCBot, Bytespider) while allowing retrieval bots.

# Allow search indexing + AI retrieval; block training scrapers only
User-agent: *
Content-Signal: search=yes,ai-train=no,use=reference
Allow: /
Disallow: /dashboard/

Disallow: /
User-agent: Bytespider
Disallow: /
User-agent: Amazonbot
Disallow: /

Disallow: /
User-agent: ChatGPT-User
Allow: /
User-agent: ClaudeBot
Disallow: /

Allow: /
User-agent: PerplexityBot
Allow: /

Allow: /

Allow: /

Technical

Invalid ItemList JSON-LD on category pages due to literal unescaped newlines

The ItemList schema block on category pages contains unescaped newline characters in the product description properties. This violates JSON standards and throws syntax parsing errors.

Page: https://beta.olympia.com.au/trophies/cricket-trophies

SEO Audit Finding

Severity: High Channels: SEOGEO

Confidence 98/100 evidence: Category page JSON-LD block

On the category pages (e.g. /trophies/cricket-trophies), the ItemList structured data block is rejected as invalid JSON by parsers. The Laravel controller outputs the product's database description field directly into the JSON string without escaping.

Because the description contains actual unescaped literal newlines (e.g., after 'retains' in the Cosmos trophy), JSON.parse crashes. Googlebot, ChatGPT and other search spiders discard the entire ItemList schema, losing category-level catalog visibility.

{
  "@type": "Product",
  "name": "Cosmos Super Mini Cricket Trophy",
  "description": "Extra size in the Cosmos series of trophies... retains 
the brilliant design elements..."
  // ❌ Literal newline after 'retains' crashes parser
}
Revamp Implementation

Use PHP's native json_encode() to serialize the structured data array rather than manually interpolating database strings into a JSON string template. This automatically escapes newlines, quotes, and backslashes.

<!-- Category Controller / Blade template -->
<script type="application/ld+json">
{!! json_encode($itemListSchemaData, JSON_UNESCAPED_SLASHES | JSON_PRETTY_PRINT) !!}
</script>

How to verify: Open Category page source, copy JSON-LD block, paste into a JSON parser (or https://jsonlint.com/) and confirm it parses with 0 errors.
